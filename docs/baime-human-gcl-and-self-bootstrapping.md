# 人类 GCL 负担与自举系统的认知经济学

**日期**：2026-06-22
**背景**：对"GCL 负担随项目自举能力上升而加重"这一现象的整体分析

---

## 核心重述：瓶颈搬家了，没有消失

自动化做的事，本质是把低信息密度的工作从人身上剥离。逐行写代码、手动派工、跑测试——这些每单位时间承载的"不可替代判断"很低。当它们被 loop-backlog / epic worker 吃掉之后，留在人手里的，是被浓缩过的、每一秒都承载大量下游后果的判断。

这产生一个反直觉的结果：

> 人的**总投入时间**可能在下降，但人的**单位时间认知密度**在飙升。

GCL（Gate Comprehension Load）测的正是后者——它是 gate 时刻的密度，不是总量。14.55 这个数字之所以重，不是因为我们让人做得更多，而是因为我们让人做的每一件事都更"满"。这是 Amdahl 定律的认知版本：优化掉可并行、可自动的部分，剩下的串行残差（人的判断）就成了纯瓶颈——没有水分可挤了。

需要补充一个重要限定：被压缩的是理解的**作用表面积**，不是认知能力的广度。局部 gate 不是低阶理解，而是全面认知能力在有限边界上的高密度投影——做出可靠判断需要同时动员目标理解、架构理解、风险判断、因果推理、对 agent 伪成功模式的经验，以及对"什么证据足够"的判断。认知门槛不是从"全局理解"降低为"局部理解"，而是从"持续执行型理解"转向"压缩判断型理解"：在更小的表面积上承载更高密度的认知判断。

**这不是做错了，这是做对了的几何必然。**

## 原理：四条硬定律的交汇

### Ashby 必备变异度定律（控制论，1956）

> 控制器的变异度必须 ≥ 被控系统的变异度，否则失控。

人是这个自指系统的 gate 控制器。当系统能力和产出速率上升（变异度上升），作为控制器的人被迫匹配同等的变异度——否则 gate 形同虚设。GCL 上升是必备变异度定律在认知层面的直接表现：**要保住控制权，就必须承担与系统复杂度同步增长的理解负担。** 要么人跟上，要么放弃控制，没有第三条免费的路。

### Bainbridge 自动化的反讽（1983）

Lisanne Bainbridge 在《Ironies of Automation》里指出：自动化把简单的部分拿走，留给人的全是困难的部分；同时它削弱人对系统的持续感知，却又要求人恰好在最困难的时刻介入决策。

baime 几乎是这篇 40 年前论文的字面实例：

- 自动化拿走了执行 → 人只剩 propose / plan / promote / confirm 这些最密的判断点
- 自治 loop 在 worktree 里跑 → 人对"系统现在长什么样"的感知反而下降（即 `proposal-situational-awareness.md` 描述的"架构断层"）
- 人被要求在 gate 上（最难的时刻）做判断，而此时他对上下文的掌握是最稀薄的

GCL 负担 = 这三个后果叠加在 gate 这一点上。

### Jevons 悖论 / 比较优势迁移（经济学）

效率提升不会减少资源消耗，而是诱发更多使用。颗粒度抬高后，单位人力能监管的在制工作量上升（loop-meta 提案明说这是目标），于是人不会"轻松了"，而是**管得更多、每个决策牵动更广**。人的比较优势从"执行"迁移到了"判断方向与守边界"——对机器最难的，正是对人最有价值的残余。

### Landauer 原理（信息热力学）

自指系统维护自我模型需要持续做功（参见 `baime-self-reference-analysis.md`"观测者负担"节）。人在 gate 上扮演 Maxwell 妖——做信息分拣，把高熵的可能性空间压成一个低熵的决策（APPROVED / ITERATE）。**这个分拣有不可消除的下界成本。** GCL 是这个妖的工作量。不能通过"更聪明"把它降到零，只能换更高效的工具帮妖干活。

## 源流：颗粒度阶梯设计出来的负担

这个负担不是从天上掉下来的，是颗粒度阶梯的台阶高度。`loop-meta-architecture.md` 的使命原话：

> "持续放大人类参与开发的颗粒度……人从逐条派工变为定目标、守边界、审方向。"

每抬高一级颗粒度——逐行 → 逐任务 → 逐 epic → 逐目标——就把更多下游后果浓缩进每一个人类决策点。这是 GCL-per-gate 上升的直接源头，而且是**刻意设计的结果**，不是意外。

```
颗粒度层级        人的 gate 时刻          浓缩的下游后果
──────────────────────────────────────────────────────
逐行            每行代码的接受            1 个文件的后续
逐任务 (现在)    task promote / DoD 验证  1 个 epic 的子路径
逐 epic         epic plan 审批            全 skill 组合演化
逐目标 (loop-meta 方向)  meta-goal 验收   整套方法论的方向
```

台阶越高，每一步看得越远、踩空的代价越大、判断密度越高。当前感到的重，是站在比以前高几级台阶上的必然。

## Gate 的局部性不等于认知的局部性

AI-native 软件工程并没有降低对人类认知能力的要求，而是改变了要求的呈现方式。

人不再需要在执行过程中连续追踪每一行代码、每一次修改、每一个中间状态——这些执行细节被 LLM agent 和自动化系统吸收。但这不意味着人类只需要狭窄的局部理解。**为了在某个 gate 点做出可靠判断，人必须动员全面的认知能力**：目标理解、系统架构理解、风险判断、因果推理、验证意识、对 agent 失效模式的经验、以及对"什么证据足够"的判断。

局部的是 gate 的作用点，不是认知能力本身。

换言之，人在 gate 上必须把全面认知压缩投影到局部判断中：在一个有限的 task、一个有限的 diff、一个有限的 DoD、一个有限的证据集上，判断它是否真的推进了系统，而不是污染了系统、绕过了验证、制造了似真伪结果，或把错误高速复制进自治执行层。

**局部 gate 理解不是低阶理解，而是全面认知能力在局部边界上的高密度投影。** "最小化所需理解表面积"的工程目标，不等于降低认知门槛，而是把全面认知的使用方式从**连续全局跟踪**改造成**高密度局部判断**。后者要求可能更高而非更低：不能把 prompt 当咒语，不能把 agent 输出当答案，不能只看测试绿灯，不能只读表面 diff；必须能在局部证据中恢复其全局后果。

## 为什么人现在仍然必要——四条结构性理由

### 1. 接地（grounding）

纯自指系统会滑向"精致地服务于开发 skill 本身却不解决外部问题"。人是那个外部锚——唯一能判断"这圈迭代到底有没有创造真实价值"的非元节点。机器无法自证接地，因为它没有系统之外的立足点。

### 2. 不动点未到，方向无法外包

OCA 的映射 `f` 还在高速移动（单月 295 commits），意味着**还没有稳定的 reward 信号可供机器对齐**。在 reward 本身还在演化的时候，"往哪走"在数学上就不可自动化——不能让系统优化一个自己还在重定义的目标。

### 3. 反 reward-hacking 的冻结锚

frozen acceptance 必须由人持有，且必须在闭环之外。一旦把验收标准放进自治回路，它就会被优化掉（TASK-52/53/54 的历史教训）。人是那个"不能被系统说服去放宽标准"的部分。

### 4. 分布外判断

自动化只在已 codify 的分布内可靠。分布外的新情况，判断权天然回到人手上。系统跑得越快，撞上分布边缘的频率越高，对人的判断的调用反而越密。

**时序区分**：理由 1 和 2 会随收敛而减弱（不动点附近 reward 稳定了、territory 探明了）；理由 3 和 4 是长期结构性的。这意味着暂态期结束后，人的必要性不会消失，但**必要性的来源会从"方向判断"迁移到"边界守护和分布外处置"**。

## 最危险的失败模式：空心 gate

需要直说一个比"人累垮"更隐蔽、更致命的失败路径：

当 GCL 超过人的实际处理能力时，**人不会停下来，人会继续盖章——但盖的是空章**。gate 表面是绿的（APPROVED），实际人已经不再真正理解他在批准什么。

这一刻，前面四条"人的必要性"全部静默失效：

- 接地锚松了但没人知道
- reward-hacking 没人挡了
- 冻结标准被悄悄侵蚀
- 分布外情况被当作分布内批准

系统看起来仍在受控运行，实际已经进入**无见证的自我修改**。这正是 `baime-self-reference-analysis.md` 里"自治 loop 改坏了自治 loop 而未被察觉"的结构性脆弱——只不过触发它的不是工程 bug，而是**人的认知预算被悄悄耗尽**。

因此 GCL 不只是舒适度指标，它是**控制权是否真实存在的指标**。当 GCL 逼近人的上限，gate 从"控制"退化为"仪式"。`proposal-situational-awareness.md` 的 GCL 压缩工作不是锦上添花，而是**维持自治系统控制合法性的基础设施**——它在守护"人的批准仍然是真批准"这件事。

## 实证：meta-cc 行为数据

以下数据来自 meta-cc 对 2026-06-20 至 06-22 三天用户输入的分析（200 条真实消息，16 个 session，主开发 session `855161fb` 独占 142 条、跨越 32 小时）。

**构念说明**：这 200 条消息是**输入行为**的总体，而非 gate 事件的总体。GCL 的 N=20 测的是 gate 时刻的理解负担；这里测的是输入模式。二者描述的是同一人在同一项目里的行为，但**不是同一个构念**——用输入行为数据来"验证"GCL 理论，是类比论证，不是实证验证。以下发现应视为与理论一致的观察，而非因果证据。

### 消息压缩：认知预算的外包策略

```
长度分布：min=3字符  median=74字符  mean=79字符  max=251字符
无一条超过 300 字符，400+ 字符：0 条
```

中位 74 字符不是风格选择，是 GCL 管理策略的直接表现：分析工作外包给 AI，自己只输入约束、判断结果、批准或拒绝。这个策略高效，但带来一个副作用——**消息越短，单字符的语义密度越高，打错的代价越大**。

最具代表性的案例：

```
queued（先提交）：删除 backlog 中 TASK ID 以 `13.` 开头的所有 task
typed（立即改正）：删除 backlog 中 TASK ID 以 `93.` 开头的所有 task
```

`13.` vs `93.`——单字符之差，如果 AI 执行了错误版本，会删除完全不同的任务集。三天内出现 6 次 queued→typed 立即更正，均是在高密度输入压力下的拼写或用词错误，被用户在毫秒到秒级内捕捉并重发。这是 GCL 压力在输入层的直接印记。

### 节奏：6.5 分钟的中位间隔

```
同 session 内消息间隔（排除 >2 小时停顿，n=180）：
  median=6.5分钟  mean=13.3分钟
  <2分钟（快速反应）：29条（16%）
  2–10分钟（主节奏）：80条（44%）
  10–30分钟（深度评估）：49条（27%）
  30分钟+（离开或深度思考）：22条（12%）
```

6.5 分钟中位间隔 = AI 执行时间 + 用户扫描结果 + 判断的总和。用户在这个节奏里不是在独立思考下一步——下一步从 AI 的输出里读出来。当 AI 走偏，出现 <2 分钟的快速连发（最多一次 17 分钟内发 4 条几乎相同的 pushback），GCL 摩擦在时间维度上是可见的。

**元分析输入的前置间隔**：7 条带有独立判断的高密度输入（"可靠是可实现的，可理解可能有内生障碍"；"我怀疑我们已经把它搞得过于复杂"；"客观地说，作为一个智人……"），实测前置停顿为 8.3、1.5、11.2、9.8、5.5、1.9 分钟——**没有一条超过 30 分钟**。"深度思考后才输出"的假说未被数据支持；元分析输入出现的时机与普通输入无显著差异。

### pushback 率：gate 是真实的

200 条消息中，33 条是明确的 pushback（"明显不符合"、"过于简单"、"注意"、"等我确认"、"先分析原因"等），**pushback 率 16.5%**。

这个数字说明 gate 不是仪式——每 6 条输入里有 1 条在纠正 AI 的质量、方向或范围。Pushback 集中在三类：AI 输出质量不达标、AI 执行范围溢出、状态/行为异常被用户发现。

### 关键推论：方差比均值更重要

上述数据揭示了一个比 GCL 均值（14.55）更重要的特征：**GCL 的方差**。

用户的信息处理模式是快速扫描 + 模式匹配 + 即时判断，而不是逐字阅读 + 独立推导。这个模式在 AI 输出格式稳定、质量一致时效率极高（6.5 分钟节奏可持续）；但当 AI 输出格式不稳定或质量参差，快速匹配失败，需要重建上下文，GCL 突然尖峰——表现为连发 pushback 或长停顿。

**最有价值的 GCL 压缩方向，因此不是让判断变少，而是降低"AI 第一次输出就建立错误上下文"的频率**——因为修复错误上下文的成本（重建 + 纠正 + 重发）远高于正常路径的 GCL。

## 工程响应：对"必备变异度"的经济应对

目标不是让人匹配系统的**全部**变异度，而是让人只匹配**不可委派的那部分**变异度。三条已有或可行的路：

**Scope−（已有实测）**：把 gate 判断范围收窄，不是降低标准，是降低单 gate 的密度。dod-eval gate 把 GCL 从 14.55 压到 5.0，差异来自判断范围，不是来自更少的约束。

**Artifact+ / 规则类外化**：把规则类隐性项（H）外化到 `docs/ARCHITECTURE.md`，让 H 转成可机械检查的 E，从而把"需要人理解"的部分变成"脚本可验证"的部分。TASK-151 的 premise-ledger 是第一步积累——先识别哪些 H 属于规则类，再逐步外化。

**分层 gate**：不是所有 gate 都需要全密度人类理解。机械 gate（DoD 条目的 shell 验证）交给脚本；判断 gate（架构演化方向、新涌现的分布外情况）留给人。把人的认知预算路由到密度真正不可压的地方。

## 三条岔路

这是价值选择，不纯是工程问题：

| 路径 | 内容 | 代价 |
|------|------|------|
| A. 压缩竞赛 | GCL 压缩工具持续跑赢能力增长，人留在环内但每秒杠杆更高 | 工具本身是 self-hosted，有自我验证盲区；人的认知余量持续被榨取 |
| B. gate 元自动化 | 把 gate 判断本身部分自动化，再加元 gate | 递归后退，元 gate 的 GCL 又上升，可能只是搬家；接地风险上升 |
| C. 主动限速 | 故意把系统增长率压到人的认知容量以内 | 放弃一部分自举速度，换取控制的真实性；适合不动点附近的收敛期 |

三条路不互斥：**暂态期适合 A**（压缩工具撑住人），**不动点附近适合 C**（主动收敛）。B 只有在 gate 判断本身已被充分量化、可对齐时才值得考虑。

## 展望：暂态假说与收敛信号

当前高负担，很可能是**快速收敛的暂态，而不是稳态**。GCL 重，是因为 `f` 还在高速移动、territory 还在被探明。当方法论逼近不动点（skill 净增量下降、变更范围收窄），新颖判断的频率会自然下降——不是因为人被自动化掉了，而是因为地图画完了，需要现场判断的未知区域变少了。

**但这个假说目前无法区分于另外两个假说**：（a）系统已经在滑向空心 gate，负担上升是"控制权正在丢失"的信号；（b）负担高不是本质复杂性的代价，而是 B″ 双状态机 / 14 列看板 / 双 daemon 等**偶然复杂性**造成的——如果架构更简单，GCL 不一定非得这么高。用户自己曾说"我怀疑我们已经把它搞得过于复杂"，这个假说不应被绕过。这两种情况需要完全不同的应对，而无法通过单次 GCL 快照来区分。

**区分它们需要把 GCL 变成时间序列。**

premise-ledger（TASK-151）已经在每次 gate 自动记账了——每个 gate 事件产生 `GCL-self-report: E=n C=n H=n`。只要让它累积，就能第一次回答：

```
GCL 在时间轴上是在下降（→ 收敛，暂态退场）
             还是持续上升（→ 能力增长吃掉压缩空间）
             还是在加速上升（→ 空心 gate 风险区，需立即干预）
```

**可操作的下一步**：

1. 以 TASK-152 为起点积累 premise-ledger 数据，每次 gate 事件记录日期 + GCL 三分量
2. 每隔 ~20 个 gate 事件做一次 GCL 趋势快照，写入 `docs/research/gcl-timeseries.md`
3. 定义两个报警阈值：
   - **收敛信号**：连续 3 个快照均值下降 ≥ 10%，且 H 分量不再增长 → 暂态退场，可适当降低 GCL 压缩投入
   - **干预信号**：连续 2 个快照均值上升 ≥ 20%，或 H / GCL 比例上升 → 暂停颗粒度抬升，优先做规则外化

这把"是否在靠近不动点"从直觉判断变成可证伪的测量，是整套自知能力（第三代，参见 `baime-self-reference-analysis.md`）的自然延伸。

## 效度威胁与证据等级

| 主张 | 证据类型 | 威胁 |
|------|---------|------|
| GCL 均值 14.55 | N=20，H 分量"中低置信度负空间估算" | 无置信区间；H 是最难测的分量，偏差方向未知 |
| dod-eval GCL=5.0 vs 全量 14.55 | 单次对照，N 未说明 | 是否显著差异未检验 |
| "消息压缩是 GCL 管理策略" | 叙事解读 | 可由"中文天然简短"、"此人话少"、任务性质解释 |
| "`13.`→`93.` 是 GCL 压力的印记" | 单个案例 | 笔误在低压场景同样发生；不可证伪 |
| pushback 率 16.5% | regex 关键词匹配，未人工校验 | "注意：中/英文输出"等格式要求被误计入 pushback |
| 行为数据"验证"了 GCL 理论 | 构念不同（输入行为 ≠ gate 事件） | 是类比论证，不是实证验证 |
| premise-ledger 可作为 GCL 时间序列 | 两种仪器（forensic 估算 vs LLM 自报）无校准对比 | 不同仪器的读数不可直接拼接；自报有动机偏差和盲区 |
| "负担是本质复杂性代价" | 四个理论框架同向支持 | 竞争假说（偶然复杂性、过度工程）未被检验 |
| "局部 gate 理解足够" | 表述推断 | 易被误读为认知门槛降低；实际是认知形式改变（连续全局追踪 → 高密度局部投影），要求可能更高而非更低 |

**本文的诚实证据等级**：定性框架（Bainbridge 反讽、颗粒度阶梯）具有解释力；行为观察（消息长度、笔误、pushback 频率）是一致性证据；GCL 数字是估算快照，非精确测量。结论应读作**待验证的假设**，而非已确立的发现。

---

## 小结

baime 的 GCL 负担，是一个健康的自举系统成功运行的副产品，而不是失败的症状。它的根源在于：自动化剥离了低密度工作，剩下的串行残差（人的判断）成了纯瓶颈；颗粒度阶梯的每次抬升都把更多下游后果浓缩进单个 gate；Ashby 定律要求控制器匹配被控系统的变异度，没有免费的路。

人的必要性是结构性的，但其来源会随收敛而迁移：暂态期是"方向判断和 reward 守护"，收敛后是"边界守护和分布外处置"。

最紧迫的任务，是不让 GCL 压力把 gate 从"控制"蚀化为"仪式"。而区分"正在收敛"和"正在失控"，只有一条路：把 GCL 快照变成时间序列，让系统自己告诉你它在往哪走。
