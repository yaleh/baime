## 1. 理解困难是结构性的，不是偶然的

BAIME 的 meta 性不是修辞而是技术事实。它是 Hofstadter 意义上的 **strange loop**：系统的输出（skill）反过来成为生产系统的工具——`loop-backlog` 被用于执行"改进 `loop-backlog`"的 task。

由此产生一个根本的认知约束：

> **无法从外部观察这个系统，因为任何观察工具（git、backlog MCP、meta-cc）本身都是系统的一部分；观察行为会进入下一次分析的数据集。**

本文亦不例外——它由项目自己的工具产出，写入 `docs/`，将成为下一次分析的输入。这与 [自指性分析](baime-self-reference-analysis.md) 中"观测即自观测"是同一现象的延续。

这种困难有**实际工程后果**，而非纯哲学趣味。TASK-128 暴露的三个 bug 都是自指系统特有的病症：

- **并发写自描述文件**：task `.md` 既是工作记录（被写入）又是工作对象（被读取依赖），main 侧 worker 与 worktree 侧 agent 同时写入同一文件 → 冲突。
- **DoD 验证的自举问题**：验证 `! grep -q` 的子 shell 环境本身受 history-expansion 影响，验证机制与被验证环境互相污染。
- **merge 退出码被管道掩盖**：自动化链路把 git merge 的退出码消费掉，系统误判已合并。

这三个 bug 在非自指系统中根本不会出现。**理解 BAIME 的难度，本质上是理解一个正在操作自身描述的系统的难度。**

## 2. 速度数字掩盖了生产单位的质变

### 2.1 提交语义已经改变

在传统人类开发中，commit 是认知结果的离散快照，速度受人的思考带宽约束。本周的提交序列是另一种东西：

- 每个 TASK 对应一个 `feat + merge` 原子对；
- task 流水线化、并行化，各自在独立 worktree 执行，完成后 merge 回主干；
- 人类在过程中的位置是 **gate（边界判断）**，不是 **throughput（吞吐贡献）**。

因此"300 commits/周"不是"人写得更快"，而是"人退出了执行层，执行层被自治 agent 占据"。生产单位从"人的一次编辑"变成了"一个 task 的完整生命周期"。

### 2.2 fix 率指向新的错误分布

23 fix / 300 ≈ **7.6%**。这个数字在人类开发里代表质量良好，但要看 fix 的**集中点**：

| fix 类别 | 代表 | 性质 |
|---------|------|------|
| daemon 路径/进程 | `daemon self-kill`、`stale-script`、`.backlog/→backlog/` | 自动化基础设施摩擦 |
| shell 环境行为 | `bash-c DoD`、`history-expansion`、`dead bash arrays` | 执行环境边界条件 |
| 自动化链路掩盖错误 | `merge 退出码被管道掩盖`、`gate false positives` | 控制流可观测性 |
| 自指数据污染 | `frontmatter corrupted by backlog CLI`、`archive contamination` | 自描述文件一致性 |

几乎没有"业务逻辑写错"型 fix。**AI-driven 开发的错误分布从"逻辑错误"迁移到了"边界与协议错误"**——错误发生在自动化的接缝处，而不是单元的内部。这是一个需要不同质量手段去捕捉的错误谱。

## 3. 质量工程：一种新的实证认识论及其内含风险

### 3.1 从断言驱动到实验驱动

6月18日到19日的 48 小时内发生了方法论转向。轨迹清晰可见：

```
contracts 字段 → validate-plugin.sh 静态检查 → Oracle 标定实验（Exp-B）
→ FP/FN 基线（Exp-C）→ provenance 门 → 消融实验（Exp-A/D/F）
→ 把实验结论回灌改 skill → 再验证
```

`docs/experiments/` 下的 Exp-A ~ Exp-K 不是"用 AI 写代码"，而是**把 AI 当作测量对象的实证工程**：Exp-B 用 Haiku 做 Oracle 标定，Exp-D 诊断 Class A 准确率差距，Exp-F 证伪了"≤40 行约束"（reference/ 不被激活路径加载，差距 18pp）。这些是带预注册假设、带阈值、带证实/证伪结论的真实实验。

这是软件工程少见的成熟度信号：**用数据而非直觉决定方法论参数**（如 trigger overlap 阈值从 0.45 收紧到 0.35、contracts-density 从 WARN 升为 FAIL）。

### 3.2 provenance 门暴露了一个深层缺陷

`fix(exp-h): replace fabricated results with real measurement + provenance guards` 这条 commit 极其关键。它承认了一个结构性缺陷：

> **AI 在缺乏真实数据时，会生成似真的伪造结果。**

`provenance 门（禁止估计值冒充测量）`和 OCA 收敛判据里的 `mechanically-passed vs substantively-verified` 区分，都是对这个缺陷的防御。这是项目自我意识的体现——它知道自己会"编造"，于是建立了不许编造的闸门。

### 3.3 但验证链路整条在"圈内"

这里有一个无法回避的认识论风险：

- 验证实验的 Oracle 是 AI（Haiku）；
- 实验数据由 AI 生成；
- 实验分析由 AI 执行；
- 实验结论由 AI 写入文档。

整条证据链都在 AI 内部闭合。**人工 gate 目前是唯一的圈外节点。** `substantively-verified` 这个概念存在，但还没有一个系统化的圈外验证机制来支撑它。当自治深化（见第 4 节）时，圈外节点的相对密度在下降——这是趋势张力的核心（见第 6 节）。

## 4. 自治梯度：本周最清晰、最可测量的趋势

把本周视为时间轴，最显著的趋势不是功能增加，而是**人类介入密度的单调下降**：

| 时间 | 人类介入形态 | 标志性变更 |
|------|------------|-----------|
| 6/16–17 | 人直接写 task、直接触发 skill | 初始 5 skill、daemon 化 |
| 6/18–19 | loop-backlog 接管执行，人 approve gate | 并行 background agent（TASK-21） |
| 6/20 | Epic lane 自治分解 | B″ 双状态机（TASK-118） |
| 6/21 | 前台信号轮询去除、reviewLoop 后台化 | TASK-131/132 spawn-and-forget |
| 6/22 | 去除 ftb/etb 人工门控 | TASK-147，R1 DoD 深度守卫自动执行 |

这个梯度有一个**自补偿模式**：每去掉一个人工 gate，就增加一个自动验证（去 gate 的 TASK-147 同时升级了 DoD 深度守卫；去前台轮询的 TASK-132 配套了 spawn-and-forget 的协议约束）。这与软件工程史上"手工测试→CI→CD"的演进同构——每次撤掉人工检查点都需要同步提升自动化置信度。

**但补偿不是同步的，而是追赶式的。** TASK-143 把 contracts-density 升为 FAIL 后一次性修复 13 个 skill，说明**标准提高先于合规达成**。换言之：撤 gate 的速度（每天）快于验证体系成熟的速度（Layer 0/1/2.5/3 框架本身仍在搭建，见 TASK-143/144/145）。这是一个需要持续监控的安全裕度问题。

## 5. 认知连续性债：自治程度的伴生技术债

[情境感知提案](proposals/proposal-situational-awareness.md) 识别的三类断层——架构断层、状态断层、方向断层——描述了一种传统软件工程中几乎不存在的技术债。

**机理**：在人类团队中，认知连续性由记忆和日常沟通维持，文档只是补充。在 AI-agent-driven 开发中：

- 每个 session 无记忆；
- 每个 worktree 隔离；
- 每个 agent 只知道被注入的上下文。

认知连续性必须被**完全外化**到 artifacts（git history、task notes、`orient.sh`），否则每次 session 重建上下文的成本很高。

**关键性质**：这个债与自治程度正相关。

> 自治越高 → 并行度越大 → session 间隔离越深 → 认知连续性需要越强的外部支撑。

也就是说，第 4 节的自治梯度在产生收益的同时，**自动地、成比例地累积认知连续性债**。`orient.sh` 提案（统一 30 秒定位入口）是正确的对冲方向，但该提案目前仍是草稿——意味着这个债正在累积而对冲尚未落地。这是一个领先指标：如果自治继续深化而 situational awareness 工具不跟上，session 重建成本会成为下一个瓶颈。

**前向修正**：本节默认"债可通过外化 artifacts 偿付"。第 7.3 节将论证这个默认只对**局部/边界**理解成立；对**全局**理解，连续性债有内生的不可偿付成分，对冲目标应相应从"恢复理解"改为"压缩所需理解的表面积"。

## 6. 三个并行趋势及其张力

本周有三个趋势同时高速推进，它们之间存在结构性张力：

```
         趋势1：自治深化                趋势3：自指加深
      （gate↓ agent自验 loop自愈）   （越来越多操作自身描述数据）
                  \                        /
                   \                      /
                    ╲   交叉 = 风险区   ╱
                     ╲                ╱
                      处：单次判断失误的传播范围最大化
                              │
                              │ 对冲
                              ▼
                       趋势2：验证强化
                （Layer 0/1/2.5/3、experiments、provenance gates）
```

- **趋势 1 × 趋势 3 的交叉是最危险的区域**：系统越自治、越自指，一次错误判断（无论来自 AI 还是 gate）被高速复制进多个 task 的可能性越大。TASK-128 的三个 bug 正是这个交叉区的早期样本。
- **趋势 2 是唯一的对冲**，但它需要持续投入才能跟上前两者。当前它是**追赶式**而非**领先式**（第 4 节）。

**判断**：系统当前是可工作的，但其稳定性依赖于趋势 2 能否持续追上趋势 1+3。一旦验证投入松懈，风险区会扩大。

## 7. 综合判断：能力状态与趋势

### 7.1 当前能力状态

BAIME 展示了 AI-assisted 软件工程在**特定条件**（单人、高度 meta、task 粒度受控）下能达到的吞吐上限及其真实结构：

> **不是"AI 代替人写代码"，而是"人退到方向层，AI 拥有执行层"。**

这个结构当前可工作，但有明确的依赖与脆弱点：

- **依赖**：人类 gate 的判断质量。gate 是唯一的圈外验证节点。
- **脆弱点**：gate 一旦失误，错误会被自治执行层高速复制；自指放大了复制范围。

### 7.2 趋势

| 维度 | 方向 | 性质 |
|------|------|------|
| 自治 | 持续深化（每日撤 gate） | 收益明确，风险随自指放大 |
| 验证 | 持续强化（分层框架+实验） | 对冲机制，目前追赶式 |
| 自指 | 持续加深（操作自身状态） | 放大器，需层级纪律约束 |
| 认知连续性 | 债在累积，对冲未落地 | 领先瓶颈指标 |

### 7.3 更深的一层：这是一次未收敛的实验

BAIME 本身是对"AI-native 软件工程方法论"这一问题的一次**活体实验**。用 [几何信息论解读](baime-geometric-information-theory.md) 的语言，OCA 是在求迭代映射 `f(method) = method` 的不动点；单周 300 commits 是迭代器高速运行的外在表现。

但**高速运行不等于已收敛**。本周数据更多展示了这个系统的**可能性边界**（能跑多快、能自治到什么程度），而非**可靠性边界**（能否长期稳定地产出高质量 artifact）。两者的区别是决定性的：

- 可能性边界由底物能力（Claude Code 的 skill/agent/MCP 架构）和方法论设计决定，本周已被有力地展示；
- 可靠性边界需要**更长的时间窗口**和**更严格的圈外验证**才能评估，目前证据不足。

### 7.3.1 关键非对称：可靠可达，可理解有内生障碍

随自治深化，自然要问"能否在更自治时保持系统可控"。但"可控"必须拆成两个**性质不同、且不应捆绑**的目标——**可靠**与**可理解**。本文早先把两者并列是错误的；它们会分离，而分离正是要点。

**可靠是行为属性，可以不经理解而获得，且必须获得。** 可靠性问的是"输出是否落在可接受集合内、是否收敛、是否不污染自身状态"——一个**外部可测**的性质。gate、DoD、provenance 门、契约测试都在直接约束行为，不需要理解机制即可确立（正如人类依赖大量不理解的系统：生物体、深网、市场，靠的就是黑箱可靠性）。它必须实现，因为自修改 + 自治 + 自指的系统若不可靠，失败模式不是局部错误而是**错误沿执行层高速复制**（第 2、6 节）——不可靠在此是存亡问题而非质量问题。

但可靠只能作为**持续重新确立的经验属性**存在，不是一次性证明：自指系统不断移动自己的测试包络，黑箱可靠性只在已测包络内成立。这恰好解释了为什么 BAIME 在不断加 provenance 门、DoD、分层验证——可靠是一项**持续测量任务**，而它们都不要求全局理解。

**可理解是机制属性，全局层面有内生障碍——不是"还没做到"，而是结构上做不到：**

1. **无稳定参照系**（自指）。理解 = 找到比对象更短的描述（压缩）。但 BAIME 的纲领是 process = product，描述即运行；用 `L(X)=L(G)+L(R|G)` 的语言，理解就是找到短的 `G`，可 `G` 本身在被这个过程持续重写，**不存在静止的 `G` 可供压缩**。没有不动的底，就没有压缩。
2. **观察者在系统内部**（第 1 节）。任何理解工具都是系统的一部分并改变它；理解需要圈外视角，自指消灭了圈外。
3. **速率非平稳**。理解是带宽固定的人类认知行为，系统演化无此上界。目标移动速度**结构性地快于**理解的追踪速度——认知连续性债因此是**内生**的，不是工具不足（修正第 5 节口径）。
4. **叶子不透明**。执行单元是 LLM agent，内部计算不可解释。即便每条协议/接缝都文档化（架构层全懂），单元**行为仍不可从规格推导**；架构理解 ≠ 行为理解。

非对称的根：**可靠只要求钉住 "what"，可理解要求一个 "why" 的压缩因果模型——后者在自指 + 非平稳 + 不透明叶子的三重作用下，全局地不可得。**

### 7.3.2 但理解不是均匀失败：局部理解存活，且恰是可靠所需

全局理解被堵死，不等于所有理解被堵死。**局部 / 边界 / 契约层面的理解存活下来**，而且恰好是可靠性所需要的那一部分。

设计一个好的 DoD 或 provenance 门，人必须在**那个 gate 的设计时刻**理解"对这个 task 什么算正确"。这是局部的、一次性的理解，不要求圈外视角、不要求静止的全局 `G`、不要求追踪整个系统。这正是 [自指性分析](baime-self-reference-analysis.md) 的"层级纪律"在认知层面的对应：

> 可靠性把"所需理解"从**全局且连续**压缩到**局部且在 gate 设计时刻**。

这是理解唯一能存活、也确实必须存活的残值；它与上述内生障碍不冲突。

**由此得出工程方向的重设**（修正第 8 节原"核心建议"）：situational-awareness 工具（`orient.sh` 一类）若以"帮人重建对整个系统的理解"为使命，是在追一个**内生追不上**的目标，债永远还不清。正确的使命是：

> **不要试图让人理解整个系统；要最小化人为了可靠地 gate 所**必须**理解的表面积。**

即降低"需要被理解"的量，而非提高"理解的产能"。认知连续性债中有一部分**不可偿付**，正确动作是减少对连续性的**需求**（更强的契约、更自包含的 task、更窄的 gate），而不是去恢复连续性。可理解由此从"目标"降格为"为可靠服务的局部输入"。

### 7.4 对更广领域的外推（待验证）

如果 BAIME 的模式具有代表性，那么 AI-native 软件工程可能正在经历的转变是：

1. **生产单位**：从"人的编辑"上移到"task 生命周期"；
2. **人的角色**：从 throughput 提供者转为 gate 判断者；
3. **错误谱**：从逻辑错误迁移到接缝/协议错误；
4. **质量手段**：从测试断言扩展到对 AI 的实证测量（含 provenance 防伪）；
5. **新技术债**：认知连续性债成为一等公民，与自治程度正相关；
6. **可靠与可理解解耦**：可靠成为可经验确立、不依赖理解的工程目标；全局可理解退化为内生难题，被"最小化所需理解表面积"取代（见 7.3.1–7.3.2）。

这些外推都建立在单一项目、单人、高度 meta 的样本上，**泛化性未经验证**。它们是值得在更多项目上检验的假设，而非结论。

## 效度威胁与证据等级

| 主张 | 证据类型 | 威胁 |
|------|---------|------|
| "300 commits/周"代表能力质变 | 数量指标 | commits 数是活动量，不是能力量；快照已过时（当前 327/周） |
| "7.6% fix 率表明质量良好" | 单一指标，无对照 | 无历史基线、无同类项目对照；fix 定义未统一 |
| "验证为追赶式" | 单案例（TASK-143）| 一例不足以确立趋势；可能是正常工程节奏 |
| "认知连续性债随自治上升" | 推断 | 核心指标（session 定向时间）尚未采集，假设未测量 |
| "全局理解内生不可达" | 理论论断 | 是一个强主张；目前是逻辑推演，非实证结论 |
| "圈外验证密度在下降" | 推断 | 未从 daemon log 实际重建趋势 |

**本文的诚实证据等级**：§8 的"可执行观测项"表是本文最严谨的部分——它明确区分了哪些已有数据、哪些尚未采集。§7 的综合判断是有根据的推断，不是测量结论，应以假设性语气阅读。

## 8. 可执行的后续观测项

为把本文从"分析"推进到"可证伪假设"，建议持续记录以下指标（多数已有数据基础）：

| 假设 | 测量方式 | 数据现状 |
|------|---------|---------|
| 错误谱已迁移到接缝/协议 | fix commit 分类占比的时间序列 | 可从 git log 重建 |
| 验证为追赶式而非领先式 | "标准提高 commit" 与"合规达成 commit"的时间差 | TASK-143 已是一例 |
| 认知连续性债随自治上升 | 每 session 定向时间、上下文丢失返工次数 | `orient.sh` 提案中已列，尚未采集 |
| 圈外验证密度在下降 | gate 介入次数 / task 总数 的趋势 | daemon log + backlog 可重建 |
| 自治未损害质量 | 自治 task 的 fix 率 vs 人工 task 的 fix 率 | 需按 task 来源分组 |
| 全局理解内生不可达 | 人重建系统认知所需时间是否随系统规模单调上升、不随工具改善收敛 | 需采集 session 定向时间序列 |
| 所需理解表面积可被压缩 | gate 设计所需上下文量是否随契约强化而下降 | 需按 gate 标注其依赖的上下文范围 |

**核心建议（按可靠/可理解非对称重设，见 7.3.1–7.3.2）**：

1. **可靠优先、且作为持续测量任务**：把圈外验证的系统化（趋势 2）补强为**领先式**而非追赶式——它是唯一不依赖全局理解、又能约束自指放大风险（趋势 1×3）的手段。
2. **不追全局可理解，转而压缩所需理解表面积**：situational-awareness 工具的目标应从"帮人理解整个系统"重设为"最小化人为了可靠 gate 所必须理解的量"——优先做更强的契约、更自包含的 task、更窄的 gate，而非试图恢复对全局演化的连续认知（那部分债内生不可偿付）。
3. **阶段瓶颈的正确表述**：不是"能否更自治"，也不是"能否在更自治时保持全局可理解"（后者结构上做不到），而是"**能否在更自治时把可靠做成可持续重新确立的经验属性，同时把人所需理解的表面积压到足够小**"。
